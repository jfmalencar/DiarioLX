\section{Overview} \label{sec:arch_overview}

The Diário LX platform follows a client-server architecture. The system is composed
of a browser-executed client application and a server-side infrastructure responsible
for business logic and data management. Figure~\ref{fig:architecture} illustrates the
high-level architecture of the system.

\begin{figure}[H]
  \begin{center}
    \resizebox{\textwidth}{!}{\includegraphics{./figures/architecture_diagram.png}}
  \end{center}
  \caption{High-level system architecture.}
  \label{fig:architecture}
\end{figure}

The system comprises the following main components:

\begin{itemize}
  \item \textbf{Frontend} (React + TypeScript): A single-page application (SPA)
        running in the browser, responsible for rendering the public website and the
        backoffice interface. Communicates with the backend via HTTP requests and
        interacts directly with SeaweedFS for media uploads and reads.

  \item \textbf{Web Service} (Spring Boot + Kotlin): The backend application
        responsible for business logic, authentication, and data access. Controls media
        metadata and mediates access to the data layer.

  \item \textbf{PostgreSQL}: Relational database storing structured data
        (content, users, categories, tags, sessions).

  \item \textbf{SeaweedFS}: Self-hosted distributed file system used for multimedia
        asset storage (images, audio, video), accessed via an S3-compatible API. Receives
        direct uploads from the frontend and is managed by the backend via media control
        operations.
\end{itemize}

\section{API Design} \label{sec:arch_api}

Communication between the frontend and backend is handled through a RESTful HTTP API.
All endpoints are prefixed with \texttt{/api/}. The API follows standard HTTP
conventions for resource naming and status codes, and all error responses follow the
Problem Details for HTTP APIs standard (\texttt{application/problem+json}), as defined
in RFC~9457~\cite{rfc9457}.

The API is organised around the following resource groups:

\begin{itemize}
  \item \texttt{/api/auth} --- Authentication (signup, login, logout, token refresh)
  \item \texttt{/api/users} --- User management (update, deactivate, delete, list)
  \item \texttt{/api/contents} --- Content management (create, edit, submit, approve, reject, publish, archive, delete)
  \item \texttt{/api/categories} --- Category management (create, edit, delete, list, archive and unarchive)
  \item \texttt{/api/tags} --- Tag management (create, edit, delete, list, archive and unarchive)
  \item \texttt{/api/medias} --- Media asset management (generate signed url, list, delete)
  \item \texttt{/api/invites} --- Invitation code generation and validation
\end{itemize}

\section{Authentication and Authorisation} \label{sec:arch_auth}

Authentication is implemented using JSON Web Tokens (JWT)~\cite{jwt_spec}. Upon
successful login, the server generates two tokens: a short-lived access token
(10 minutes), signed with HMAC-SHA256, and a long-lived refresh token (7 days),
which is a cryptographically random opaque string. Both are issued as
\texttt{HttpOnly} cookies, preventing client-side JavaScript from accessing them
and mitigating XSS attacks.

The access token encodes the user's identifier and role as JWT claims, enabling
stateless validation on every request without a database lookup. Refresh tokens
are persisted in the \texttt{sessions} table and invalidated upon logout, providing
explicit session termination. When the access token expires, the client transparently
obtains a new one via \texttt{POST /api/auth/refresh}.

Role-based access control (RBAC) is enforced at the HTTP layer, with each endpoint
declaring the minimum required role. The implementation details are covered in
Chapter~\ref{cap:backend}.

Content transitions between the following states:

\begin{description}
  \item[Draft] Initial state. The content is only visible to its author.
  \item[Pending Approval] The author submits the content for review. It becomes
        visible to Editors and Administrators.
  \item[Published] An Editor or Administrator approves the content; it becomes
        publicly accessible.
  \item[Rejected] An Editor or Administrator rejects the content, optionally
        providing revision comments. The author may edit and resubmit.
\end{description}

Editors and Administrators may also publish content directly, bypassing the
approval step.
