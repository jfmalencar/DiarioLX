\chapter{Architecture} \label{cap:architecture}

This chapter presents the architecture of \textit{Diário LX}. It gives a high-level view of
the system and its main components, the technology stack that realises them, and the design
of the REST API, and then examines the cross-cutting concerns that shape the system: its
authentication and authorization model, the way media is stored and served, and the
editorial workflow that governs a content item's lifecycle.

\section{Overview} \label{sec:arch_overview}

The \textit{Diário LX} platform follows a client-server architecture. The system is composed
of a browser-executed client application and a server-side infrastructure responsible
for business logic and data management. Figure~\ref{fig:architecture} illustrates the
high-level architecture of the system.

\begin{figure}[H]
      \begin{center}
            \resizebox{\textwidth}{!}{\includegraphics{./figures/architecture_diagram.png}}
      \end{center}
      \caption{High-level system architecture.}
      \label{fig:architecture}
\end{figure}

The system comprises the following main components:

\begin{itemize}
      \item \textbf{Frontend} (React + TypeScript): A single-page application (SPA)
            running in the browser, responsible for rendering the public website and the
            backoffice interface. Communicates with the backend via HTTP requests and
            uploads media directly to the object store, bypassing the application server.

      \item \textbf{Web Service} (Spring Boot + Kotlin): The backend application
            responsible for business logic, authentication, and data access. Controls media
            metadata and mediates access to the data layer.

      \item \textbf{PostgreSQL}~\cite{postgresql_docs}: Relational database storing structured data
            (content, users, categories, tags, sessions).

      \item \textbf{SeaweedFS}~\cite{seaweedfs}: Self-hosted distributed file system used for multimedia
            asset storage (images, audio, video), accessed via an API compatible with Amazon S3~\cite{amazon_s3}. Receives
            direct uploads from the frontend and is managed by the backend via media control
            operations.
\end{itemize}

\section{API Design} \label{sec:arch_api}

Communication between the frontend and backend is handled through a
RESTful HTTP API, following the architectural principles described by
Fielding~\cite{fielding2000rest} and the practical conventions of
Richardson and Amundsen~\cite{richardson2013restful}. All endpoints are prefixed with \texttt{/api/}.
The API follows standard HTTP conventions for resource naming and status codes,
and all error responses follow the Problem Details for HTTP APIs standard
(\texttt{application/problem+json}), as defined in RFC~9457~\cite{rfc9457}.

The API is divided into two groups: a public, unauthenticated read API that
serves the reader-facing website, and an authenticated \textit{backoffice} API,
prefixed with \texttt{/api/backoffice/}, that provides editorial management
operations. This separation makes the authentication boundary explicit in the URL
structure and mirrors the frontend's \texttt{/backoffice} route.

Authentication is handled separately under \texttt{/api/auth}, which provides the
signup, login, logout, and token refresh endpoints used to establish and maintain
user sessions.

The public read API is organized around the following resources:

\begin{itemize}
      \item \texttt{/api/homepage} --- homepage composition (the section menu is delivered through the bootstrap/settings payload)
      \item \texttt{/api/contents} and \texttt{/api/contents/\{slug\}} --- listing and retrieval of published content
      \item \texttt{/api/categories/\{slug\}} and \texttt{/api/tags/\{slug\}} --- content grouped by category or tag
      \item \texttt{/api/team} and \texttt{/api/authors/\{slug\}} --- the editorial team and individual author pages
\end{itemize}

The backoffice API groups the management operations, each guarded by the minimum
required role:

\begin{itemize}
      \item \texttt{/api/backoffice/contents} --- content management (create, edit, submit, publish/approve, reject, archive, unarchive, delete, history)
      \item \texttt{/api/backoffice/categories} and \texttt{/api/backoffice/tags} --- category and tag management (create, edit, delete, list, archive, unarchive)
      \item \texttt{/api/backoffice/medias} --- media asset management (upload, list, generate signed URL)
      \item \texttt{/api/backoffice/users} --- user management (list, update, account status, team assignment)
      \item \texttt{/api/backoffice/invites} --- invitation code generation and validation
      \item \texttt{/api/backoffice/featured} --- curation of the homepage featured content
      \item \texttt{/api/backoffice/settings} --- site configuration
\end{itemize}

\section{Authentication and Authorization} \label{sec:arch_auth}

Authentication is implemented using JSON Web Tokens (JWT)~\cite{jwt_spec}. Upon
successful login, the server generates two tokens: a short-lived access token
(10 minutes), whose integrity and authenticity are protected by an HMAC-SHA256
message authentication code (MAC), and a long-lived refresh token (7 days),
represented as a cryptographically random opaque string. Both are issued as
\texttt{HttpOnly} and \texttt{SameSite=Strict} cookies --- and additionally with the
\texttt{Secure} attribute when served over HTTPS in production --- preventing
client-side JavaScript from accessing them (mitigating XSS), restricting their
inclusion in cross-site requests (mitigating CSRF), and, in production, ensuring
they are only transmitted over encrypted connections. While access tokens are validated statelessly,
refresh tokens are persisted in the database to support session invalidation,
logout, and propagation of user role changes.

The access token encodes the user's identifier and role as JWT claims, which is what
makes that validation stateless, while refresh tokens live in the \texttt{sessions}
table. When the access token expires, the client transparently obtains a new one via
\texttt{POST /api/auth/refresh}.

Role-based access control (RBAC)~\cite{sandhu1996rbac} is enforced at the HTTP layer, with each endpoint
declaring the minimum required role. The implementation details are covered in
Chapter~\ref{cap:backend}.

\section{Media Storage} \label{sec:arch_media}

Content in \textit{Diário LX} is rich in multimedia --- images, audio, and video --- and
media files can be large, with uploads permitted up to two gigabytes. Routing such
files through the application server would make it a bottleneck: each upload would
occupy server memory and connection time merely to relay bytes it does not otherwise
need to process. To avoid this, media is kept in a dedicated object store, separate
from the relational database, and the transfer of the file itself is decoupled from the
application server.

The store is SeaweedFS, accessed through an interface compatible with Amazon S3.
Choosing an S3-compatible interface keeps the system independent of any single storage
provider: the self-hosted SeaweedFS deployment used in this project could be replaced by
a managed object store, or by any other S3-compatible service, without changes to the
application. This isolation is reflected in the backend, where storage is defined as an
abstraction with an interchangeable implementation, as described in
Chapter~\ref{cap:backend}.

Crucially, the media bytes never pass through the application server, in either direction.
On upload, the browser obtains a short-lived (fifteen-minute) authorization from the
backend and transfers the file directly to the object store; the backend never handles the
file contents, only recording the media's metadata --- its location, type, and descriptive
fields such as caption and credit. Delivery mirrors this: readers' browsers load images,
audio, and video directly from the object store through the URLs kept in that metadata.
Keeping the server off the byte path leaves it responsive under large uploads and heavy
read traffic alike. The concrete mechanism is detailed in Chapter~\ref{cap:backend}.

\section{Editorial Workflow} \label{sec:arch_workflow}

The editorial workflow governs the lifecycle of a content item from creation to
publication. It enforces editorial accountability by requiring explicit approval
before any content becomes publicly accessible, while allowing Editors and
Administrators to intervene at any stage. Each content item carries a state,
represented in the domain by the \texttt{ContentState} enumeration, which takes one
of four values: \textit{Draft}, \textit{Pending Review}, \textit{Approved}, or
\textit{Rejected}. Figure~\ref{fig:workflow} summarises the states and the
transitions between them, each annotated with the role required to perform it.

\begin{figure}[H]
      \begin{center}
            \includegraphics[width=0.85\textwidth]{./figures/workflow_diagram.png}
      \end{center}
      \caption{Editorial workflow: content states and the role-guarded transitions between them.}
      \label{fig:workflow}
\end{figure}

A deliberate design decision separates the editorial \emph{approval} of a content
item from its \emph{publication}. Approval is the moderation act by which an Editor or
Administrator endorses the content and is captured by the \textit{Approved} state,
whereas publication is the moment the item actually becomes visible to readers and is
governed by a separate \texttt{published\_at} timestamp. An item is exposed publicly
only when it is \textit{Approved} \emph{and} its \texttt{published\_at} instant has
been reached. Keeping these two concerns distinct lets the editorial decision be
recorded independently of when the content goes live, and in particular enables
publication to be scheduled for a future instant.

The states are defined as follows:

\begin{description}
      \item[Draft] Initial state. The content is created and edited by its author and
            is not publicly visible. A Contributor works on a draft until all required
            fields are complete and then submits it for review.

      \item[Pending Review] The author has submitted the content for editorial review.
            It now awaits evaluation by an Editor or Administrator, who either approve it
            for publication or reject it with feedback.

      \item[Approved] An Editor or Administrator has approved the content. From this
            point the item is eligible for publication; it becomes publicly accessible
            once its \texttt{published\_at} instant is reached, which may be immediate or
            scheduled for a later time.

      \item[Rejected] An Editor or Administrator has rejected the content, providing a
            reason. The reviewer and the rejection comment are retained in the
            content's history, so that the author receives structured feedback and can
            revise and resubmit.
\end{description}

The transitions between these states are the following:

\begin{itemize}
      \item \textbf{Submit} --- a Contributor submits a draft, moving it from
            \textit{Draft} to \textit{Pending Review}.

      \item \textbf{Approve} --- an Editor or Administrator approves a pending item,
            moving it to \textit{Approved}; the reviewer identity is recorded and a
            \texttt{published\_at} instant is set, either immediately or scheduled for a
            future time.

      \item \textbf{Reject} --- an Editor or Administrator rejects a pending item,
            moving it to \textit{Rejected} together with a review comment.

      \item \textbf{Direct approval} --- because Editors and Administrators hold the
            approval permission, they may approve a draft directly, bypassing the
            \textit{Pending Review} state when editorial review is not required.

      \item \textbf{Edit} --- editing a \textit{Rejected} or \textit{Approved} item
            returns it to \textit{Draft}. A rejected item is revised by its author and
            resubmitted; an approved item can only be edited by an Editor or
            Administrator.
\end{itemize}

These transitions are enforced at the API level. Each is exposed as a dedicated
endpoint under \texttt{/api/backoffice/contents} and guarded by the
\texttt{@RequireRole} annotation, so the backend validates that the requesting user's
role permits the transition before persisting the state change. The permission model
and the implementation of these checks are described in Chapter~\ref{cap:backend}.
