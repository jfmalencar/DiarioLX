\chapter{Frontend} \label{cap:frontend}

\section{Technology} \label{sec:fe_technology}

The frontend is developed as a single-page application (SPA) using \textbf{React}
with \textbf{TypeScript}. React's component-based model facilitates the construction
of reusable UI elements and enables clear separation between the public website and
the backoffice interface. TypeScript was adopted to improve code maintainability and
reduce runtime errors through static type checking.

The following libraries were adopted:

\begin{itemize}
      \item \textbf{React Router}~\cite{react_router} --- client-side routing, enabling
            slug-based navigation.

      \item \textbf{Fetch API}~\cite{mdn_fetch} --- HTTP communication with the backend
            API, wrapped in a custom \texttt{useApi} hook.

      \item \textbf{TipTap}~\cite{tiptap} --- rich text editor framework used to implement
            the block-based article editor in the backoffice.

      \item \textbf{Bootstrap}~\cite{bootstrap} --- responsive CSS framework used for
            layout structuring and interface styling.

      \item \textbf{Lucide React}~\cite{lucide} --- icon library providing consistent
            and lightweight SVG-based icons throughout the interface.
\end{itemize}

\section{Organization} \label{sec:fe_organization}

The frontend follows a modular architecture inspired by the separation of concerns
principle. Figure~\ref{fig:frontend_diagram} illustrates the internal organization
of the frontend and its relationship with the backend API.

\begin{figure}[H]
      \begin{center}
            \resizebox{\textwidth}{!}{\includegraphics{./figures/frontend_module_diagram.png}}
      \end{center}
      \caption{Frontend module organization.}
      \label{fig:frontend_diagram}
\end{figure}

The source tree is organized as follows:

\begin{verbatim}
src/
  app/
    providers/        # React context providers (auth, bootstrap)
    App.tsx           # Root component and router setup
  assets/             # Static assets (images, fonts)
  config/             # Application configuration
  layouts/            # Shared page layout components
  modules/
    backoffice/       # Backoffice-specific pages and components
    public/           # Public website pages and components
  shared/
    components/       # Reusable UI components
    hooks/            # Custom React hooks
    services/         # API communication layer
    types/            # TypeScript type definitions
    utils/            # Utility functions
  tests/              # Playwright end-to-end tests
\end{verbatim}

The main modules are:

\begin{description}
      \item[UI Components] Reusable presentation components with no business logic,
            shared across the backoffice and public modules.

      \item[Hooks] Custom React hooks encapsulating state management and side effects.
            Hooks communicate with the backend exclusively through the \texttt{Services}
            interface.

      \item[Services] An interface defining the API communication contract, implemented
            by the \texttt{API Client}. This abstraction decouples the application logic
            from the HTTP communication layer.

      \item[API Client] The concrete implementation of the \texttt{Services} interface,
            built on the Fetch API and encapsulated in the \texttt{useApi} hook.

      \item[Domain] TypeScript type definitions representing the shared data model
            (content, users, categories, tags).
\end{description}

\section{Bootstrap and Configuration} \label{sec:fe_bootstrap}

On application startup, the \texttt{useBootstrap} hook fetches the initial
configuration from the backend, including the API endpoint definitions. This avoids
hardcoding API URLs in the frontend codebase, allowing the backend base URL to be
configured via environment variables without recompiling the frontend.

\section{Navigation} \label{sec:fe_navigation}

Client-side routing is handled by React Router. URL slugs follow the pattern
\texttt{/<category>/<slug>}. Category and tag listing pages are dynamically
generated based on data fetched from the API.

\section{Authentication} \label{sec:fe_auth}

Authentication state is managed via a React context provider, which stores the
authenticated user's identity and role after a successful login. Since tokens are
stored in \texttt{HttpOnly} cookies and managed by the browser, the frontend does
not handle token storage directly.

The \texttt{useApi} hook wraps all API calls and handles token expiry transparently:
when a request returns \texttt{401 Unauthorized}, the hook automatically attempts a
token refresh via \texttt{POST /api/auth/refresh}. If the refresh succeeds, the
original request is retried; otherwise, the user is redirected to the login page.
To avoid duplicate refresh requests when multiple concurrent calls expire
simultaneously, a shared \texttt{refreshPromise} ensures that only one refresh
attempt is made at a time.

Protected routes redirect unauthenticated users to the login page.

\section{Backoffice} \label{sec:fe_backoffice}

The backoffice is the editorial management interface, accessible only to
authenticated users. It provides tools for user administration, category and tag
management, and invitation-based onboarding. Content management is partially
implemented — content creation is functional, but editing, deletion, and the
editorial review workflow are currently under active development.

\subsection{Authentication Pages} \label{sec:fe_auth_pages}

Access to the backoffice is restricted to invited users. The authentication flow
comprises two pages: a login page for returning users and a registration page for
new users who have received an invitation code.

The registration form collects the user's first and last name, username, email,
password, and invitation code. The invitation code is validated server-side and
determines the role assigned to the new account. Both pages share a split layout
--- a full-height cover image with the Diário LX branding on the left, and the
form on the right.

\section{Public Website} \label{sec:fe_public}

The public website is the reader-facing interface of Diário LX, providing access
to published content organized by editorial section. It is designed to reflect the
editorial identity of the publication, prioritising long-form reading and visual
journalism. At the current beta stage, only the homepage and article page are
implemented; the remaining pages are planned for the final version.

\subsection{Homepage} \label{sec:fe_homepage}

The homepage has been implemented and currently presents content from multiple

Photography, Podcasts, and Videos. At the current beta stage, the page is

\section{Responsive Design} \label{sec:fe_responsive}

The interface is built on Bootstrap's responsive grid system, adapting layouts
to mobile, tablet, and desktop viewports through standard Bootstrap breakpoints.
Responsive design is currently in an initial phase, with full coverage planned
for the final version.

\section{Tests} \label{sec:fe_tests}

Frontend end-to-end testing is implemented using Playwright~\cite{playwright}, a
browser automation framework that supports testing across Chromium, Firefox, and
WebKit. Playwright tests simulate real user interactions against the running
application, covering the main workflows of both the public website and the
backoffice. The test suite is currently in an early phase and is integrated into
the continuous integration pipeline via GitHub Actions~\cite{github_actions},
running automatically on every push.
