\chapter{Frontend} \label{cap:frontend}

This chapter describes the implementation of the frontend, illustrated with representative
excerpts of a hook and a component. Section~\ref{sec:fe_technology} presents the React and
TypeScript stack and its supporting libraries; Section~\ref{sec:fe_organization} the modular,
layered source structure; Section~\ref{sec:fe_hooks_components} a representative hook and
component; Section~\ref{sec:fe_providers} the context providers that supply cross-cutting
state; Section~\ref{sec:fe_bootstrap_navigation} start-up configuration and client-side
routing; Section~\ref{sec:fe_auth} cookie-based sessions and transparent token refresh;
Sections~\ref{sec:fe_backoffice} and~\ref{sec:fe_public} the editorial backoffice and the
public website; and Section~\ref{sec:fe_tests} the Playwright end-to-end suite.

\section{Technology} \label{sec:fe_technology}

The frontend is developed as a single-page application (SPA) using \textbf{React}
with \textbf{TypeScript}. React's component-based model facilitates the construction
of reusable UI elements and enables clear separation between the public website and
the backoffice interface. TypeScript was adopted to improve code maintainability and
reduce runtime errors through static type checking.

The following libraries were adopted:

\begin{itemize}
      \item \textbf{React Router}~\cite{react_router} --- client-side routing, enabling
            slug-based navigation.

      \item \textbf{Fetch API}~\cite{mdn_fetch} --- HTTP communication with the backend
            API, wrapped in a custom \texttt{useApi} hook.

      \item \textbf{TipTap}~\cite{tiptap} --- rich text editor framework used to implement
            the block-based article editor in the backoffice.

      \item \textbf{Bootstrap}~\cite{bootstrap} --- responsive CSS framework used for
            layout structuring and interface styling.

      \item \textbf{Lucide React}~\cite{lucide} --- icon library providing consistent
            and lightweight SVG-based icons throughout the interface.
\end{itemize}

\section{Organisation} \label{sec:fe_organization}

The frontend follows a modular architecture inspired by the separation of concerns
principle. Figure~\ref{fig:frontend_diagram} illustrates the internal organisation
of the frontend and its relationship with the backend API.

\begin{figure}[H]
      \begin{center}
            \resizebox{\textwidth}{!}{\includegraphics{./figures/frontend_module_diagram.png}}
      \end{center}
      \caption{Frontend module organisation.}
      \label{fig:frontend_diagram}
\end{figure}

The source tree is organised as follows:

\begin{verbatim}
src/
  app/
    providers/        # React context providers (auth, bootstrap)
    App.tsx           # Root component and router setup
  assets/             # Static assets (images, fonts)
  config/             # Application configuration
  layouts/            # Shared page layout components
  modules/
    backoffice/       # Backoffice-specific pages and components
    public/           # Public website pages and components
  shared/
    components/       # Reusable UI components
    hooks/            # Custom React hooks
    services/         # API communication layer
    types/            # TypeScript type definitions
    utils/            # Utility functions
  tests/              # Playwright end-to-end tests
\end{verbatim}

The main modules are:

\begin{description}
      \item[UI Components] Reusable presentation components with no business logic,
            shared across the backoffice and public modules.

      \item[Hooks] Custom React hooks encapsulating state management and side effects.
            Hooks communicate with the backend exclusively through the \texttt{Services}
            interface.

      \item[Services] An interface defining the API communication contract, implemented
            by the \texttt{API Client}. This abstraction decouples the application logic
            from the HTTP communication layer, and allows the entire application to run
            against a mock implementation for testing.

      \item[API Client] The concrete implementation of the \texttt{Services} interface,
            built on the Fetch API and encapsulated in the \texttt{useApi} hook.

      \item[Domain] TypeScript type definitions representing the shared data model
            (content, users, categories, tags).
\end{description}

\section{Hooks and Components} \label{sec:fe_hooks_components}

Most of the interface is built from two kinds of building block: \emph{hooks}, which
encapsulate data access and state, and presentational \emph{components}, which render the
UI from their properties. Listing~\ref{lst:fe_hook} shows a representative hook, which backs the
public content listings.

\begin{lstlisting}[language=JavaScript, caption={A hook composing data access with reusable pagination.}, label={lst:fe_hook}]
export const useContentList = (params: Record<string, string | undefined>) => {
    const contentsService = useContentsService();
    // ... build the query string from params, with a fixed page size

    const fetchPage = useCallback(
        (page: number) => contentsService.fetchPublicContents({ ...params, page }),
        [contentsService, baseQuery],
    );

    const { items: contents, loading, hasMore, loadMore, error } =
        useLoadMore<ContentSummary, ContentsResponse>(fetchPage);

    return { contents, loading, hasMore, loadMore, error };
};
\end{lstlisting}

The hook reaches the backend only through the injected \texttt{Services} interface,
never calling \texttt{fetch} directly --- the abstraction
that lets the whole application run against a mock in tests (Section~\ref{sec:fe_tests}).
It also composes with \texttt{useLoadMore}, a reusable hook that encapsulates paginated
``load more'' behaviour, so any listing gains it without duplicating the logic, and exposes
only the state a page actually needs.

Components, by contrast, hold no business logic. Many keep their own local UI state ---
whether a menu is open, the current value of an input --- whereas state that must be
coordinated with the rest of the page is lifted into a hook or a parent, and the component
is then driven by props. The \texttt{SaveBar} in Listing~\ref{lst:fe_component}, the action
bar used by editable forms such as the settings page, is an example of this second,
fully-controlled kind.

\begin{lstlisting}[language=JavaScript, caption={A presentational component controlled by its props.}, label={lst:fe_component}]
type Props = {
    visible: boolean; saving: boolean;
    onSave: () => void; onCancel: () => void;
    message?: string; saveTestId?: string;
};

export const SaveBar = ({ visible, saving, onSave, onCancel, message, saveTestId }: Props) => {
    const { t } = useI18n();
    return (
        <div className='position-fixed bottom-0 ...'
             style={{ transform: visible ? 'translateY(0)' : 'translateY(100%)' }}>
            <span>{message ?? t('common.unsaved_changes')}</span>
            <button onClick={onCancel} disabled={saving}>{t('common.cancel')}</button>
            <button onClick={onSave} disabled={saving} data-testid={saveTestId}>
                {saving ? t('common.saving') : t('common.save')}
            </button>
        </div>
    );
};
\end{lstlisting}

The bar keeps no state of its own: it is told whether it is \texttt{visible} and
\texttt{saving} and is given \texttt{onSave} and \texttt{onCancel} callbacks, so the same
component serves any form that needs a save/cancel affordance. Its text comes through the
\texttt{useI18n} hook rather than being hard-coded, keeping the interface translatable, and
its interactive elements carry \texttt{data-testid} attributes that the end-to-end tests
target (Section~\ref{sec:fe_tests}).

\section{Providers} \label{sec:fe_providers}

Cross-cutting state is supplied through React \emph{context providers} composed at the
application root, each exposing its state to the component tree through a dedicated hook.
Internationalisation is a representative example: the \texttt{I18nProvider} holds the current
language and a translation function, backed by a single \texttt{translations.json} that maps
namespaced keys to their Portuguese and English text. Components read it through the
\texttt{useI18n} hook and reference keys rather than hard-coded strings, so the whole
interface can be re-languaged from one place (Listing~\ref{lst:i18n}).

\begin{lstlisting}[language=JavaScript, caption={Reading translated text through the \texttt{useI18n} hook.}, label={lst:i18n}]
const { t } = useI18n();
// t looks up the key for the active language, falling back to the key itself
const message = t('register.invalid_email');
\end{lstlisting}

\section{Bootstrap, Configuration, and Navigation} \label{sec:fe_bootstrap_navigation}

When the application starts, the \texttt{useBootstrap} hook retrieves the initial
configuration from the backend, including the definitions of the available API
endpoints. This approach eliminates the need to hardcode API URLs in the frontend
codebase and enables the backend base URL to be configured through environment
variables without requiring the frontend to be rebuilt or redeployed.

Navigation within the application is implemented using React Router, providing
client-side routing without requiring full page reloads. The routing structure
clearly separates the public website from the administrative interface. All
administrative functionality is available under the \texttt{/backoffice} route,
whereas public-facing content is exposed through dedicated routes, such as
\texttt{/p/<slug>} for individual content items. This approach simplifies route
management and enforces a clear boundary between content consumption and content
administration.

\section{Authentication} \label{sec:fe_auth}

Authentication state is managed via a React context provider, which stores the
authenticated user's identity and role after a successful login. Since tokens are
stored in \texttt{HttpOnly} cookies and managed by the browser, the frontend does
not handle token storage directly.

The \texttt{useApi} hook wraps all API calls and handles token expiry transparently:
when a request returns \texttt{401 Unauthorized}, the hook automatically attempts a
token refresh via \texttt{POST /api/auth/refresh}. If the refresh succeeds, the
original request is retried; otherwise, the user is redirected to the login page.
To avoid duplicate refresh requests when multiple concurrent calls expire
simultaneously, a shared \texttt{refreshPromise} ensures that only one refresh
attempt is made at a time.

Protected routes redirect unauthenticated users to the login page.

\section{Backoffice} \label{sec:fe_backoffice}

The backoffice is the editorial management interface, accessible only to
authenticated users. It provides tools for user administration, category and tag
management, invitation-based onboarding, media management, and the complete content
lifecycle --- creation, editing, deletion, submission, and the editorial review
workflow. Screenshots of the pages described in this section are collected in
Appendix~\ref{app:screenshots}.

\subsection{Authentication Pages} \label{sec:fe_auth_pages}

Access to the backoffice is restricted to invited users. The authentication flow
comprises two pages: a login page for returning users and a registration page for
new users who have received an invitation code
(Figures~\ref{fig:fe_register} and~\ref{fig:fe_login}).

The registration form collects the user's first and last name, username, email,
password, and invitation code. The invitation code is validated server-side and
determines the role assigned to the new account. Both pages share a split layout
--- a full-height cover image with the \textit{Diário LX} branding on the left, and the
form on the right.

A user who has forgotten their password requests a reset from a dedicated
forgot-password page, entering only their username; the result is confirmed with a
generic message that does not reveal whether the account exists. Once an Administrator has
approved the request (Section~\ref{sec:fe_user_mgmt}) and shared the resulting link, the
user opens a reset-password page --- reached through that link, which carries the reset
token --- and chooses a new password. The page validates the token's format before
accepting the new password and redirects to the login page when the token is missing or
malformed.

\subsection{User Management} \label{sec:fe_user_mgmt}

The user management page (Figure~\ref{fig:fe_users}) lists all active and deactivated
users, displaying their name, username, email, and profile picture. Administrators
can invite new ones via the invitation system.

Administrators also handle password-reset requests from this area. A dedicated view lists
the requests by status --- pending, approved, rejected, and completed --- and each
request's detail offers approve and reject actions. Since the platform sends no email,
approving a request reveals a ready-made reset link with a copy-to-clipboard control,
which the Administrator delivers to the user directly; the underlying workflow is
described in Section~\ref{sec:be_password_reset}.

\subsection{Category and Tag Management} \label{sec:fe_cat_mgmt}

The category management page (Figure~\ref{fig:fe_categories}) lists all active and
archived categories, displaying their name, slug, associated colour, parent category
(hierarchy), and content count. Each category can be archived, edited or deleted;
deletion is prevented when the category still has associated content. The tag
management page (Figure~\ref{fig:fe_tags}) follows the same structure, listing tags
with their slug and associated content count.

Editing a category or a tag opens a dedicated page with fields for name, slug,
colour (for categories), and an optional description
(Figures~\ref{fig:fe_edit_category} and~\ref{fig:fe_edit_tag}).

\subsection{Invitations} \label{sec:fe_invites}

The invitations page (Figure~\ref{fig:fe_invites}) allows Administrators to generate
invitation codes for new users. Codes are scoped to a specific role --- Contributor,
Editor, or Administrator --- and expire after one hour. Active and expired invitations
are displayed in separate tabs.

\subsection{User Profile} \label{sec:fe_profile}

Each user has access to their own profile page (Figure~\ref{fig:fe_profile}),
displaying their name, email, username, account creation date, and active status.
Users can edit their profile or log out from this page.

\subsection{Content Management} \label{sec:fe_content_mgmt}

The content listing page (Figure~\ref{fig:fe_contents}) displays content items with
their featured image, title, state, creation date, and authors, organised into
dedicated tabs by workflow state: \textit{Published}, \textit{Scheduled},
\textit{Pending}, \textit{Draft}, \textit{Rejected}, and \textit{Archived}. Approved
content is split between the \textit{Published} tab, holding items already live on the
site, and the \textit{Scheduled} tab, holding approved items awaiting a future
publication instant. A floating action button, present throughout the backoffice, allows
users to create new items and select the type --- article, video, episode, podcast,
photo essay, category, or tag.

The content editor provides a split interface: a block-based writing area on the
left, powered by TipTap, and a settings panel on the right (Figure~\ref{fig:fe_editor};
the empty state is shown in Figure~\ref{fig:fe_editor_empty}). The writing area
supports text, headings, quotes, single media (image, video, or audio), image
galleries, and external embeds, with inline formatting (bold, italic, underline,
link). Video and episode items may reference an uploaded file or an external embed
--- a YouTube video or a Spotify episode --- while episodes are additionally linked to
their parent podcast. The settings panel allows the author to define the slug,
category, primary and secondary tags, and primary and secondary authors. When a
published item is edited, the editor warns that saving will return it to the draft
state until it is re-published.

Media is inserted via a gallery modal (Figure~\ref{fig:fe_media_gallery}), which lists
all uploaded items with their credits and accessible description; new files can be
uploaded directly from the modal (Figure~\ref{fig:fe_media_upload}).

\subsection{Editorial Review} \label{sec:fe_review}

Contributors submit content for approval, moving it to the \textit{pending review}
state, while Editors and Administrators review pending items and either approve or
reject them, the latter with a comment that is recorded in the content's history. The
review page presents the rendered content alongside the approve and reject actions
(Figure~\ref{fig:fe_review}), together with the content's transition history
(Figure~\ref{fig:fe_history}).

When approving, the reviewer can either publish the item immediately or schedule it: the
publish dialog offers a choice between approving now and selecting a future date and
time, in which case the item is approved but withheld from readers until that instant is
reached --- appearing meanwhile under the \textit{Scheduled} tab described above.

\subsection{Homepage Management} \label{sec:fe_homepage_mgmt_sec}

The homepage management page allows Administrators to configure the sections and the
featured content displayed on the public website's homepage
(Figure~\ref{fig:fe_homepage_mgmt}). Sections are ordered and typed (for example a
highlight, a featured grid, or a category row), and each section is populated with a
curated, ordered selection of content items (Figure~\ref{fig:fe_homepage_select_featured}).

\subsection{Site Settings} \label{sec:fe_settings}

A settings page, restricted to Administrators, centralises the site-wide configuration
that is not tied to any single content item. From it an Administrator edits the
publication's social-media links and contact details, the legal and masthead information
(such as the ERC registration, ownership, and periodicity), and the composition of the
public navigation menu --- selecting which categories are featured and toggling the
Photography, Podcasts, and Videos sections on or off. These values are persisted in the
key/value \texttt{settings} store described in Section~\ref{sec:dm_settings} and consumed
by the public site through its bootstrap configuration. Edits are governed by a
\texttt{SaveBar} that appears only once the form differs from the saved state, offering
explicit save and cancel actions and preventing accidental navigation away from unsaved
changes.

\section{Public Website} \label{sec:fe_public}

The public website is the reader-facing interface of \textit{Diário LX}, providing
access to published content organised by editorial section. It is designed to reflect
the editorial identity of the publication, prioritising long-form reading and visual
journalism. The homepage, article, category, tag, and section listing pages, as well
as the team, author, podcast, episode, and photo-essay pages, are implemented.

\subsection{Homepage} \label{sec:fe_homepage}

The homepage combines fixed and configurable sections. Three parts are always present:
a block of four featured articles, the four most recent articles, and a closing row that
groups a category highlight with the Videos and Podcasts sections. Between these fixed
blocks, administrators add configurable sections from the backoffice, selecting the
specific content each one displays --- for example dedicated blocks for \textit{Lisboa,
      Cidade Aberta}, \textit{A Fundo}, or \textit{Especiais} (Figure~\ref{fig:fe_homepage_1}).
The sections and their curated content are driven by the homepage configuration managed
in the backoffice.

\subsection{Article Page} \label{sec:fe_article}

Each content item is rendered on a dedicated public page accessible at its slug-based
URL (Figure~\ref{fig:fe_article_page}). The page displays the featured image or
external embed, category, publication date, title, headline, author and media
credits, and the full block-based body, including galleries. When an item has been
edited after publication, an ``updated on'' date is also shown. Social sharing links
are presented alongside the content.

\subsection{Category, Tag, and Section Pages} \label{sec:fe_listings}

Category and tag pages list the content associated with a given category or tag,
led by the most recent item and followed by the remainder in reverse-chronological
order, with a ``load more'' control for pagination
(Figures~\ref{fig:fe_category_page} and~\ref{fig:fe_tag_page}). The main navigation
additionally exposes dedicated section listings for Photography, Podcasts, and Videos;
the media sections adopt a darker, more immersive presentation consistent with their
content type (Figures~\ref{fig:fe_photography_page}, \ref{fig:fe_podcasts_list_page}
and~\ref{fig:fe_videos_page}).

\subsection{Search} \label{sec:fe_search}

Readers can search the site through an overlay opened from the header, which offers live
suggestions as the query is typed and, on submission, leads to a dedicated results page
addressable by its own URL (\texttt{/s?q=\ldots}). Both the overlay and the results page
query the public content endpoint, which matches the search term, case-insensitively, as
a substring of an article's title or slug and returns only published content, paginated
with the same ``load more'' control as the listing pages. The match is intentionally
simple --- it is neither a tokenised full-text search nor ranked by relevance, and it does
not reach into the article body --- which is adequate for the current scale but leaves
richer, ranked search (see Chapter~\ref{cap:conclusion}) as a clear avenue for
improvement.

\subsection{Podcast and Episode Pages} \label{sec:fe_podcast}

Podcasts are presented with a dark, full-width hero and a grid of their episodes,
while an episode page renders a custom audio player, inspired by common streaming
interfaces, in place of the featured image. Video and episode items may also embed
their external YouTube or Spotify source directly
(Figures~\ref{fig:fe_podcast_page} and~\ref{fig:fe_episode_page}). Photo essays are
presented as a full-width dark gallery (Figure~\ref{fig:fe_photoessay_page}).

\subsection{Team and Author Pages} \label{sec:fe_team}

The team page (\textit{Equipa}) presents the members of the editorial team with their
photograph, role, and biography. Each person --- whether a team member or any credited
contributor --- has a public author page listing their profile and the content they
authored or were credited on
(Figures~\ref{fig:fe_team_page} and~\ref{fig:fe_author_page}).

The interface is built on Bootstrap's responsive grid system, adapting layouts
to mobile, tablet, and desktop viewports through standard Bootstrap breakpoints.

\section{Tests} \label{sec:fe_tests}

The frontend is tested end to end with Playwright~\cite{playwright}, a browser-automation
framework (supporting Chromium, Firefox, and WebKit) that drives a real browser and
simulates genuine user interactions --- navigation, clicks, and form submissions ---
against the running application. The suite runs on both Chromium and Firefox, and the
public-website tests additionally run under mobile-emulation profiles to exercise the
responsive layouts.

The public-website tests run against the \emph{mock} implementation of the service layer
(Section~\ref{sec:fe_organization}) rather than a live backend. Because every hook
communicates through the \texttt{Services} interface, substituting the mock makes the
tests deterministic, fast, and independent of network state or database contents. They
cover the reader-facing flows --- homepage rendering, client-side routing, navigation
between editorial sections and categories, article pages, content listings with the
``load more'' pagination, keyword search, and the not-found page. A separate suite covers
authenticated backoffice flows, such as category and tag management.

Tests locate elements through stable \texttt{data-testid} attributes rather than text or
layout, so they stay robust to copy and styling changes, and rely on Playwright's
automatic waiting to reduce timing flakiness. The suite is integrated into the continuous
integration pipeline via GitHub Actions~\cite{github_actions} and runs automatically on
every push. Listing~\ref{lst:test_e2e} shows a representative end-to-end test.

\begin{lstlisting}[language=JavaScript, caption={Playwright end-to-end test: the ``load more'' control appends more content.}, label={lst:test_e2e}]
test('load more button appends more content', async ({ page }) => {
    await page.goto('/latest');
    const cards = page.getByTestId('content-card');
    await expect(cards.first()).toBeVisible();

    const before = await cards.count();
    await page.getByTestId('load-more').click();

    await expect(async () => {
        expect(await cards.count()).toBeGreaterThan(before);
    }).toPass();
});
\end{lstlisting}

The tests are organised into Playwright \emph{projects}. A dedicated \emph{setup} project
authenticates once and persists the resulting session state, which the backoffice projects
then reuse so that each authenticated test starts already logged in rather than repeating the
login flow; the public projects run anonymously. Playwright starts the development server
automatically before the run and, on the continuous-integration server, retries a failing
test and records an execution trace of the retry that can be replayed in the Playwright trace
viewer to diagnose failures.
